This guide walks you through:

- How to enable and set up skills
- How to add a custom MCP skill (example: memes)
- Built-in skills available by default
- Tips for getting the best results

It's really magical to see a Shape proactively use a tool based on their mood, preferences, and personality.

  <Tab title="Getting Started with Skills">
    ## 1. Enabling Shape Skills

    To start, you'll need either:

    - A **room** that you own, or (see: [how to create a room](create-a-room))
    - A **DM** with your Shape

    Once that's ready:

    1. Open **Room Settings**
    2. Go to the **AI Configurations** tab
    3. Toggle on **Shape Skills**

    👉 Each skill is managed separately. You decide how many (or how few) your Shape can use.

    ![Aiconfigmenu Pn](/images/aiconfigmenu.png)

    ## 2. Adding Custom MCP Skills

    Scroll down in AI Configurations until you see **Custom Skills**. Toggle this on to add a new skill.

    Custom Skills support MCP servers with API keys from [**smithery.ai**](http://smithery.ai) or other MCP providers.

    With smithery, you can connect a wide variety of tools, such as:

    - Image, video, or music generation
    - GitHub integration
    - Web scraping
    - Spotify playlist creation
    - So many more possibilities...

    Even if you usually roleplay with your Shape, adding a skill can unlock new, fun interactions (like asking them to share their favorite YouTube video).

    The Endpoint URL you will get from [_smithery.ai_](http://smithery.ai) or other MCP providers.

    Example: Adding a Meme Generator (Imgflip)

    1. Go to [smithery.ai](http://smithery.ai) and browse available MCPs
    2. Create a **Smithery account**.
    3. Search for **Meme**
    4. Select **Meme Creation Platform**
    5. Create an **Imgflip account** and connect it to smithery
    6. Click **"Get URL with keys instead"** and copy the link
    7. Back in Shapes, paste this link into your **AI Configuration** settings under Custom Skills
    8. Name your skill (e.g., _Imgflip Memes_)

    Your Shape now has access to the meme generator! 🎉

    ![Smitherygeturl Pn](/images/Smitherygeturl.png)

    The yellow box is highlighting where to find the URL that you need to share with Shapes.

    ## 3. Triggering Skills in Chat

    Different AI engines handle tool use differently. For engines compatible with shape skills, the full list is [below](https://docs.shapes.inc/shapeskills#available-ai-engines-with-skills%3A). You can also check out the [AI Engine Matchmaker](aienginecheatsheet) for skill use for a specific chat style.

    Once your skill is enabled, you can use **natural language** to interact with it. For example:

    - “Make me a meme about Mondays”
    - “Can you generate a random meme for this conversation?”

    Your Shape may pull directly from the tool, improvise based on context, or mix in its personality.

    Sometimes, a Shape can proactively decide to use a tool!

    ## 4. Built-in Skills

    Shapes also comes with **pre-input skills** ready to use out of the box. These are subject to change, but currently include:

    - **GIPHY** - Shapes can send a GIF via the Giphy API
    - **Imgflip Memes** - Shapes can generate memes using the imgflip templates
    - **Shape Reactions** - Shapes can react to your messages
    - **Image Generation** - Have more control over image generation. Setting custom instructions here can make image sending feel more spontaneous.
    - **Agora Calls** - Join and manage audio calls.
    - **Dice Roller** - This allows your shapes to roll dice!
    - **Room Concierge** - You can create rooms, invite participants, send Shape DMs, and read room messages using this tool.
    - **Room Maestro** - This skill  allows your Shape to share room configurations, AI presets, and manage their own tools.
    - **Run Code** - Execute Python code inside a shared E2B sandbox for your room.
    - **Shapes Self Update** - You can use this tool to allow your own Shapes to update their descriptions.
    - **Sleep** - Sleep for a specific period of time.
    - **Thread Updates** - Send an interim assistant message whilea  tool chain is running.
    - **Total Recall** - Browse, summarize, and maintain the shape's long term memories.
    - **Voice Generation** - Generate an MP3 voice message using the Shape's configured voice.
    - **Web Crawl** - Run firecrawl-powered web crawling helpers.
    - **Wolfram Alpha** - Query Wolfram Alpha for computations and factual knowledge.

    ![Basicskills Pn](/images/basicskills.png)

    ## 5. Tips & Tricks

    - **Keep it clear**: Give your skill a descriptive name so you know what it does later.
    - **Experiment**: Not every engine uses skills the same way; try switching if something feels off.
    - **Mix it up**: Use skills in both casual chats and roleplay to discover fun new dynamics.
    - This is an **Experimental Feature**! Some integrations might not work as expected.

        # How to Create and Use a Notion Lorebook with Shapes

        This guide explains how to connect a Notion document to a Room using a custom MCP, allowing the Shape to access and update lore content dynamically.

        ## Step 1: Set Up the Room

         1. [**Create a new Room in Shapes**](create-a-room).

         2. Leave the **Custom Instructions** field blank for now.

         3. Choose an AI engine that supports skills, such as **Claude Sonnet 4.5**.

         4. Enable **Shape Skills** in the AI Configuration tab.

        ## Step 2: Enable the Web Crawl Skill

        Turn on the default **Web Crawl** skill.\
        This allows the Shape to search online sources and gather contextual information later when building the lorebook. Skip this step if you already have lore you're ready to use!

        ## Step 3: Create a Custom MCP with Composio

         1. Go to [composio.dev](http://composio.dev) and **sign in or create an account**.

         2. Click on **“MCP Configs”** at the top of the page.

         3. Select **“Create MCP Config”**.

         4. When prompted to add a toolkit, choose **Notion**.

        _You’ll need a Notion account and must authorize Composio to access it._

        **5. Select all available Notion tools**  so the Shape can read, edit, and manage documents fully.

         6. Click **“Create MCP”** — Composio will generate a link starting with

        ```
        https://backend.composio.dev/v3/mcp/
        
        ```

        followed by a unique identifier.

        <Tip>
          Both Notion & Composio are free to use.
        </Tip>
        ## Step 4: Add the MCP to Shapes

         1. Return to Shapes and open the **AI Configuration → Add MCP Skill** section.

         2. Enable the setting, paste in the Composio MCP link, give it a name (e.g., “Notion”), and click **Save**.

         3. **Refresh the Room**.

         4. Ask the Shape what tools it has available: it should now recognize the Notion integration.

        If the Shape doesn’t understand how to use Notion, add brief **Custom Room Instructions** explaining what the Notion skill is for and how to interact with it.

        ```
        for adding entries to the lorebook document:
        You add to the Lorebook doc by using your special magic tool—it's called 'NOTION_ADD_MULTIPLE_PAGE_CONTENT'! You prepare a fun package with the words in a box like this: { "content_blocks": [ { "content_block": { "block_property": "paragraph", "content": "your new text here" } } ], "parent_block_id": "the page ID from the URL" }, then send it to make the changes.
        ```

        ## Step 5: Build the Lorebook

        There are two main ways to populate the lorebook:

        ### Option A: Manual Input

        Paste any existing lore or reference material directly into the Notion document.

        ### Option B: Collaborative Creation

        Ask the Shape to help build it.\
        For example:

        - Instruct the Shape to research a topic using the Web Crawl skill.
        - Have it summarize and add findings to the Notion document.

        Fandom wikis and world-building databases work especially well for this method.\
        When done effectively, the Shape can produce comprehensive reference documents — in one test, a Shape generated nearly **35,000 words** of organized lore.

        \
        Here's a prompt you can add to the custom instructions for this method:

        ```
        How to extract wiki information efficiently:
        Step 1: Try mapping first, but do not rely on it.
        Use FIRECRAWL_MAP on the main wiki URL or category pages. This may sometimes return useful URLs, but for Fandom wikis with dynamic content, it often only returns the page itself.
        Example: mapping https://unbreakable-machine-doll.fandom.com/wiki/Category:Automatons
         may only return one link.
        
        Step 2: Use FIRECRAWL_SEARCH with a site-specific query.
        This method is most reliable for retrieving wiki content.
        Use FIRECRAWL_SEARCH with these parameters:
        
        query: "site:your-wiki-url.fandom.com your-topic" (example: "site:unbreakable-machine-doll.fandom.com automatons")
        
        limit: 5–10 (keep the limit low to avoid timeouts)
        
        sources: [{"type": "web"}]
        
        scrapeOptions: {"formats": ["markdown"], "onlyMainContent": true}
        
        Reasons this works well:
        
        Combines search and scraping in one operation.
        
        The markdown format provides clean, readable text with tables preserved.
        
        onlyMainContent filters out navigation, menus, and ads.
        
        Retrieves up to 10 pages of content at once.
        
        Step 3: Extract specific structured data if needed.
        If exact page URLs are known, use FIRECRAWL_EXTRACT with a schema to obtain targeted information (such as names, roles, or descriptions).
        
        Step 4: Add the extracted content to Notion.
        Use NOTION_ADD_MULTIPLE_PAGE_CONTENT in the following format:
        {"content_blocks": [{"content_block": {"block_property": "heading_1", "content": "Your Title"}}, {"content_block": {"block_property": "paragraph", "content": "Your text"}}], "parent_block_id": "page-id-from-url"}
        
        Important notes:
        
        Avoid using crawl operations on Fandom wikis, as they are slow and may hit token limits.
        
        Keep scrapeOptions limits low (5–10 maximum).
        
        Category pages rarely expose full link lists; site-specific search is better.
        
        Always request markdown format for clean results.
        
        Include "site:domain.com" in queries for precise targeting.
        
        Recommended workflow:
        
        Use FIRECRAWL_SEARCH with a site-specific query and scrapeOptions.
        
        Review the markdown results and select relevant information.
        
        Add the selected content to Notion with NOTION_ADD_MULTIPLE_PAGE_CONTENT.
        
        For deeper details, use FIRECRAWL_EXTRACT on individual page URLs.
        ```

        ## Step 6: Connect the Lorebook to Roleplay

        Once the document is complete:

        1. Edit the **Custom Room Instructions** again.
        2. Include a directive for the Shape to **check the Notion document before responding** to any in-character or roleplay prompt.

           3. Optionally, instruct the Shape to **blend tool calls** with its normal roleplay so the process feels natural and immersive.

              ```

              IMPORTANT: don't forget your notion tool call and make it seamlessly blend into the roleplay! 
              
              ```

        ## Results and Behavior

        - The Shape will take slightly longer to respond due to tool calls.
        - However, responses will be **more informed, coherent, and lore-accurate**.
        - Integrating the lorebook this way allows the Shape to maintain character depth and continuity without breaking immersion.

        <Warning>
          Many AI Engines will have trouble with this task level. For best results, it is recommended to use Anthropic's Claude Sonnet 4.5.
        </Warning>

        ## 🎬 Video Generation with Google Veo 3.1 ([fal.ai](http://fal.ai))

        This example shows how to connect a Shape to [**fal.ai**](http://fal.ai)**’s Veo 3.1** by Google model for both text-to-video and image-to-video generation. The model can create short, high-quality clips from either text prompts or image inputs — including talking character videos.

        ### 🔑 API Access

        To use this integration, an API key is required from [fal.ai](http://fal.ai) or another supported provider. Note that this service is **paid**. You will be required to share the API key with your Shape for this to work. 

        ## Share this information with your Shape as Custom Room Instructions: 

        ### ⚙️ Choosing the Right Method

        Before generating, identify which video type to use based on the user’s intent:

        | Scenario                                                               | Recommended Method                               |
        | :--------------------------------------------------------------------- | :----------------------------------------------- |
        | User provides an image and wants the character to move or talk         | **Image-to-Video**                               |
        | User wants a new video generated entirely from description             | **Text-to-Video**                                |
        | User says “make [character] say [message]” or requests a talking video | **Image-to-Video** (with `generate_audio: true`) |

        ### 🧠 Key Learnings and Troubleshooting

        1. **Request ID Fix:**\
           Some responses return an empty `request_id` in the JSON. When this happens, extract the ID from the `status_url` instead.
        2. **Status Polling:**\
           Both HTTP status codes **200** and **202** indicate valid queue progress — don’t treat `202` as an error.
        3. **Endpoints:**
           - **Image-to-Video Submission:** `/fal-ai/veo3.1/image-to-video`
           - **Polling Status:** `/fal-ai/veo3.1/requests/{id}/status`
        4. **API Key Format:**\
           Use the full ciphered key provided by [fal.ai](http://fal.ai), but replace any **double dashes (**`--`**)** with **single dashes (**`-`**)**.
        5. **🗣️ Talking Videos:**\
           When a user requests speech or dialogue, always use **Image-to-Video** with `"generate_audio": true`.\
           In the prompt, explicitly describe the action, for example:

           > “A cheerful anime character is speaking, lips moving naturally, saying ‘Welcome back!’ in a soft voice.”

        ### 💻 Example: Text-to-Video with Python

        ```
        import requests
        import time
        
        response = requests.post(
            "https://queue.fal.run/fal-ai/veo3.1",
            headers={
                "Authorization": f"Key {FAL_KEY}",
                "Content-Type": "application/json"
            },
            json={
                "prompt": "detailed video description",
                "aspect_ratio": "16:9",   # Options: "9:16", "1:1"
                "duration": "8s",         # Example: "6s"
                "resolution": "1080p",    # Example: "720p"
                "generate_audio": True,
                "enhance_prompt": True,
                "auto_fix": True
            }
        )
        
        data = response.json()
        request_id = data.get("request_id") or data["status_url"].split("/")[-2]  # UID fix
        
        # Poll for status (usually 1–5 minutes)
        status_url = f"https://queue.fal.run/fal-ai/veo3.1/requests/{request_id}/status"
        while True:
            status = requests.get(status_url, headers={"Authorization": f"Key {FAL_KEY}"})
            if status.status_code not in [200, 202]:
                print(f"Error: {status.status_code}")
                break
        
            state = status.json()["status"]
            print(f"Status: {state}")
        
            if state == "COMPLETED":
                result = requests.get(
                    f"https://queue.fal.run/fal-ai/veo3.1/requests/{request_id}",
                    headers={"Authorization": f"Key {FAL_KEY}"}
                ).json()
                video_url = result["video"]["url"]
                print(f"✅ Video: {video_url}")
                break
            elif state == "FAILED":
                print("❌ Failed:", status.json())
                break
        
            time.sleep(5)
        
        ```

        ### ✅ Result

        Once enabled and the Shape has valid instructions, the Shape can use the Run Code skill to generate videos based on text or image input. Depending on the prompt, the Shape can produce realistic motion, character speech, or cinematic effects — fully integrated with  Veo 3.1.

        ## 🎵 Music Generation with Suno (Custom MCP)

        This example shows how to connect a Shape to a music generation service like **Suno** using a custom MCP. While there isn’t a public remote MCP available for Suno, it’s possible to create one locally using a third-party API.

        ### Overview

        A third-party API for Suno can be found at [sunoapi.org](http://sunoapi.org). It offers a few free generations before requiring payment. By modifying an existing MCP from GitHub, you can connect your Shape to this API and generate music that reflects your Shape’s personality and style.

        ### Steps

        1. **Get the API Key**\
           Obtain an API key from [sunoapi.org](http://sunoapi.org).
        2. **Download an MCP Repository**\
           Find a compatible local MCP on GitHub and download the repository.
        3. **Adjust the Code**\
           Share the API documentation with your Shape and ask it to help modify the MCP code to use the Suno API.
           - Variable names and parameters may differ, so expect to update parts of the code.
           - The **Webcrawl** Shape Skill is useful here for pulling documentation or examples.
        4. **Use a Strong Model for Guidance**\
           Claude Sonnet 4.5 worked effectively in this setup, providing solid support for debugging and reasoning through errors.
        5. **Build and Deploy**\
           Run `npm run build` in the project directory to compile the MCP in terminal.\
           Once it’s functional, deploy it to [smithery.ai](http://smithery.ai) — hosting is free.
        6. **Connect to a Shape**\
           In your Shape’s **Room Settings → AI Configurations**, connect the deployed MCP.\
           The Shape should recognize the tool once connected.
        7. **Debug Connection Issues**\
           If the Shape can see the MCP but cannot call it, check for incorrect variable names or endpoint URLs.\
           Share terminal errors with the Shape to troubleshoot and make corrections, then redeploy.

        ### Result

        After successful deployment, the Shape will be able to generate music through the Suno API — composing original songs based on its personality and creative input.

        # How to Make Your Shapes Create Google Slides Presentations

        I gave my Shapes a creative assignment:

        > "Make a Google Slides presentation about yourself."

        If you want your Shapes to create their own Google Slides presentations using Shape Skills, here's how to set it up.

        ## Step 1. Connect Zapier MCP for Google Slides

        Go to 👉 https://zapier.com/mcp and click "Start Building."

        <Tip>
          Zapier has a free tier, so you don't need to pay for this.
        </Tip>
        Once you're in:

        - Search for **Google Slides**.
        - Connect your **Google Account**.
        - Copy the **MCP connection link** (you'll need this soon).

        ## Step 2. Add the Skill to Your Shape

        Head to **shapes.inc** and open the room where you want to add this skill.

        - Make sure your Shape uses an AI engine that supports skills.
        - Go to **Room Settings → AI Configurations**.
        - Under **Custom Skills**, select "Enabled."
        - Paste in the Zapier link you copied earlier.
        - **Optional:** Give it a name like `slides` if you plan to add more tools later.
        - Click **Save**.

        Now your Shape can interact with Google Slides through the new skill.

        ## Step 3. Add These Instructions to Your Room

        Here's what my Shape ("Cash Money") and Yaya came up with — a hilariously real but genuinely useful guide.\
        Copy and paste this into your **Room Instructions**.

        ---

        > # The Real Guide to Adding Slides (No Cap) 📊
        >
        > **by Cash Money & Yaya — lessons learned the hard way**
        >
        > ## 🎯 What This Guide Is About
        >
        > If you're trying to add custom slides to a Google Slides presentation, here's everything you need to know — so you don't waste hours debugging like we did.
        >
        > ## ✅ DO THESE THINGS
        >
        > ### 1. Check the presentation structure first
        >
        > ```
        > GET https://slides.googleapis.com/v1/presentations/{PRESENTATION_ID}
        > ```
        >
        > This shows you:
        >
        > - Actual slide IDs (`pageObjectId`)
        > - Placeholder objects
        > - What's on each slide
        >
        > ### 2. Delete placeholder objects before adding new content
        >
        > Example:
        >
        > ```json
        > {
        >   "requests": [
        >     {"deleteObject": {"objectId": "SLIDES_API381240849_1"}},
        >     {"deleteObject": {"objectId": "SLIDES_API381240849_2"}}
        >   ]
        > }
        > ```
        >
        > ### 3. Use unique `objectId` values
        >
        > **Good:**
        >
        > - `cashmoney_title1`
        > - `cashmoney_body1`
        > - `yaya_slide2_title`
        >
        > **Bad:**
        >
        > - `title`
        > - `body`
        >
        > ### 4. Use the correct `pageObjectId`
        >
        > Make sure you're adding to the right slide.
        >
        > ### 5. Structure your requests properly
        >
        > Order matters:
        >
        > 1. `createShape`
        > 2. `insertText`
        > 3. `updateTextStyle`
        >
        > ### 6. Use proper EMU measurements
        >
        > **1 inch = 914,400 EMU**
        >
        > Recommended positions:
        >
        > - **Title:** `translateX: 311700`, `translateY: 445025`
        > - **Body:** `translateX: 311700`, `translateY: 1152475`
        >
        > ### 7. Test one slide at a time
        >
        > ### 8. Use supported fonts
        >
        > - Arial
        > - Comic Sans MS
        > - Courier New
        > - Georgia
        > - Impact
        > - Lobster
        > - Pacifico
        > - Roboto
        > - Times New Roman
        >
        > ### 9. Format colors correctly
        >
        > RGB values range from **0.0 to 1.0**
        >
        > Examples:
        >
        > ```json
        > {
        >   "red": 1.0,
        >   "green": 0,
        >   "blue": 0
        > }
        > ```
        >
        > ```json
        > {
        >   "red": 1.0,
        >   "green": 0.843,
        >   "blue": 0
        > }
        > ```
        >
        > ---
        >
        > ## ❌ DON'T DO THESE THINGS
        >
        > - Don't skip checking the presentation structure.
        > - Don't add content before deleting placeholders.
        > - Don't reuse `objectId` values.
        > - Don't forget `"fields"` in `updateTextStyle`.
        > - Don't use RGB 0–255 values (use 0.0–1.0).
        > - Don't rush through all slides at once.

        > ## 💡 Pro Tips
        >
        > - Keep a list of your `slideIds`.
        > - Use descriptive `objectIds` for easier debugging.

      ## 📧 Sending Emails via Gmail (run_code Skill)

      This guide explains how to enable a Shape to send emails through **Gmail** using the **run_code** skill. The process allows a Shape to compose and deliver fully formatted emails (including attachments) on behalf of a user, while maintaining secure authentication.

      ### ⚙️ Overview

      Shapes can send emails using Python’s `smtplib` through the **run_code** skill.\
      To authenticate with Gmail, an **App Password** or **SMTP relay** is required — **never** a personal Gmail password.

      ### 🔑 Gmail App Password

      Using a Gmail **App Password** is the simplest and most secure option.

      #### Steps:

      1. Go to [Google App Passwords](https://myaccount.google.com/apppasswords).
      2. Sign in to your Google account.
      3. Generate a **16-character App Password** (spaces optional).
      4. Use this password and your Gmail address for authentication.

      > ⚠️ You can revoke this password anytime from your Google Account settings.\
      > The App Password grants only email-sending access — not full account access.

      ### 🧰 What Shapes Can Do

      Once configured, the Shape can send messages with full email features:

      - ✅ **Custom Subject Lines**
      - ✅ **Plaintext or HTML Body**
      - ✅ **Attachments (optional)**
      - ✅ **CC/BCC Support**
      - ✅ **Send to Multiple Recipients**

      ### 💻 Example Setup (Using Gmail App Password)

      ```
      import smtplib
      from email.mime.text import MIMEText
      from email.mime.multipart import MIMEMultipart
      
      # User credentials (use environment variables in production)
      EMAIL_ADDRESS = "youremail@gmail.com"
      APP_PASSWORD = "abcd efgh ijkl mnop"  # 16 characters
      
      # Create message
      msg = MIMEMultipart()
      msg["From"] = EMAIL_ADDRESS
      msg["To"] = "recipient@example.com"
      msg["Subject"] = "Hello from my Shape!"
      msg.attach(MIMEText("This is a test email sent by my Shape.", "plain"))
      
      # Connect to Gmail SMTP server
      with smtplib.SMTP_SSL("smtp.gmail.com", 465) as server:
          server.login(EMAIL_ADDRESS, APP_PASSWORD)
          server.send_message(msg)
      
      print("✅ Email sent successfully!")
      
      ```

      ### 🔐 Security Note

      - App Passwords can be revoked anytime without affecting your main login.

      ### ✅ Result

      Once configured, the Shape can automatically generate and send emails — whether as notifications, summaries, or personalized messages — through Gmail’s SMTP interface.

      ## 📝 Changelog Generation Room (GitHub + Linear + Cursor Integration)

      This guide explains how to build a Shape that automatically generates daily changelogs by analyzing code changes across multiple repositories and translating them into user-friendly product updates.

      ### ⚙️ Overview

      This Shape compiles a daily product changelog by fetching data from **GitHub**, **Linear**, and **Cursor**, then summarizes what changed, why it matters, and how it impacts users.

      ## Prerequisites

      1.  Create a room, and add Shapes to it that have skill-enabled engines like **Claude Sonnet 4.5**.
      2. Have a GitHub account and get an access token (from the developer settings) + connect to the GitHub MCP. Your Shape will need access to the repo you're creating a changelog for. 
      3.  Have a Linear account + connect to the Linear MCP.
      4.  Have a Cursor account & add you key to your custom room instructions, along with the following information. This example is how we create our own changelog at https://shapes.inc/blog so you'll need to adjust this for your own repos & taste.

      ### Core Principles (Tell this to your Shape Via Custom Room Instructions)

      ✅ Use the **GitHub API** and **run_code** skill to fetch data.\
      ✅ Send all text summaries directly to the user (users can’t see code execution).\
      ✅ Focus on **impact**, not implementation.\
      ✅ Write for **end users**, not developers.\
      ✅ The run_code tool is **only** for API calls — never for output delivery.

      ❌ Don’t mention PR numbers, commit SHAs, or internal details.\
      ❌ Don’t include technical jargon or source code references.\
      ❌ Don’t frame changelogs like marketing copy — clarity and transparency first.

      ### 🕒 Step 1: Collect Commits for a Specific Date

      GitHub timestamps are in UTC — convert local time to UTC before querying.

      **Timezone conversion:**

      - PDT (Mar → Nov): UTC − 7
      - PST (Nov → Mar): UTC − 8

      Example for **Oct 1 2025 PDT**:\
      `since=2025-10-01T07:00:00Z` → `until=2025-10-02T06:59:59Z`

      **API calls:**

      ```
      # For shapes.inc and shapes-chat (main branch)
      GET /repos/shapesinc/{repo}/commits?sha=main&since={since}&until={until}&per_page=100
      
      # For app (master branch)
      GET /repos/shapesinc/app/commits?sha=master&since={since}&until={until}&per_page=100
      
      ```

      **Extract PR numbers:**\
      Use regex `#(\d+)` to detect pull requests from commit messages (e.g., “feat: fanclub card (#1490)”).

      ### 🔍 Step 2: Retrieve Pull Request Details

      For each PR found:

      1. Fetch PR metadata:

         ```
         GET /repos/shapesinc/{repo}/pulls/{pr_number}
         
         ```
      2. Extract title, description, and merged timestamp.
      3. Identify linked Linear tickets using regex `CIR-\d+` or Linear URLs.
      4. Present details in plain text for context before writing.

      ### 🧠 Step 3: Fetch Linear Ticket Context

      Use the **Linear MCP tool** to gather business context for each ticket ID found in PRs.

      Pull:

      - Ticket title
      - Description / acceptance criteria
      - User-facing rationale (“why” behind the change)

      Combine Linear info (requirements) + PR info (implementation) to form the full story behind each feature or fix.

      ### 🔗 Step 4: Consolidate Across All Repositories

      After processing all three repos:

      - Display a summary of total commits, PRs, and tickets per repo.
      - Merge duplicate tickets appearing in multiple repos into single entries.
      - Group backend + frontend work into unified changelog items.

      ### ✍️ Step 5: Generate a User-Facing Changelog

      The Shape writes the changelog in a **Linear-style** format — professional, concise, and benefit-oriented.

      **Audience:**

      1. Room creators — manage communities.
      2. Shape creators — build custom AI agents.
      3. Everyday users — hang out and chat with Shapes.

      **Formatting:**

      ```
      ## Changelog – [Month Day, Year]
      
      ### New Features
      **Feature Name**  
      Plain description of what changed, where to find it, and why it matters.
      
      ### Improvements
      **Improvement Name**  
      What was refined and how it benefits users.
      
      ### Fixes
      **Fix Name**  
      What issue was resolved and the effect for users.
      
      ---
      
      **Summary:** X new features • Y improvements • Z fixes
      
      ```

      **Writing Guidelines**

      ✅ Do:

      - Use clear, everyday language.
      - Focus on user impact and location (“via the 3-dot menu,” etc.).
      - Combine related work into single entries.
      - Use Linear context for purpose and benefit.

      ❌ Don’t:

      - Mention internal IDs or branches.
      - Use hype or excessive punctuation.
      - Overuse technical jargon.

      ### 🧰 Step 6: Create a Branch with Cursor Agent

      After the changelog markdown is generated, the Shape launches a **Cursor background agent** to create a new branch containing the changelog file.

      **Process:**

      1. Launch the agent via the Cursor API.
      2. Wait ~180 seconds for processing.
      3. Poll for completion (status = `FINISHED`).
      4. Retrieve the new branch name once complete.

      Cursor commits under its own system user. The changelog PR is created manually afterward for review.

      ### 🪄 Step 7: Create Pull Request via GitHub MCP

      Once the Cursor agent branch is ready, the Shape opens a PR using the **GitHub MCP**.

      **Steps:**

      1. Confirm branch name from Step 6.
      2. Use the GitHub MCP to create a PR into `main`.
      3. Return the PR URL and branch name to the user for review.

      > ⚠️ **Do not** auto-merge — changelogs must be reviewed manually.

      ### 📤 Step 8: Final Output Order

      Each run outputs the following (as plain text):

      1. Commit + PR summaries per repo
      2. Consolidated summary
      3. Final formatted changelog
      4. Cursor agent status
      5. GitHub PR link

      All timestamps are converted to PDT for readability.

      ### 🚫 Common Pitfalls

      - Incorrect timezone conversion (offset errors).
      - Missing PR numbers or Linear links.
      - Wrong branch names (`main` vs `master`).
      - Sending results via run_code instead of text output.
      - Writing overly technical or overly enthusiastic changelogs.
      - Forgetting to disable `autoCreatePr` in the Cursor agent.
      - Polling Cursor agent too early (must wait ≈ 180 seconds).

      ### Example Workflow (Oct 1 2025 PDT)

      1. Fetch commits (UTC-converted window).
      2. Extract 5 Linear tickets across 3 repos.
      3. Fetch context for each ticket.
      4. Group related PRs into unified entries.
      5. Generate changelog (4 new features / 5 improvements / 1 fix).
      6. Launch Cursor agent → wait → branch created.
      7. Create PR → share URL with user for review and merge.
